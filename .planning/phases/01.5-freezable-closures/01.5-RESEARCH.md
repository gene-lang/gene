## Summary

Phase 1.5 should extend the existing Phase 1 freeze substrate to `VkFunction` closures by validating and tagging the closure's captured environment, not by inventing a second send or serialization mechanism. The current runtime already stores closure capture through `Function.parent_scope` and reconstructs call scopes from that pointer on invocation, so the missing work is a deep-freeze rule for captured `Scope` graphs plus tests that prove pointer-sharing is safe once a closure is frozen. (`src/gene/types/type_defs.nim:613-647`, `src/gene/vm/exec.nim:3424-3452`, `src/gene/vm/exec.nim:1754-1761`) [VERIFIED: codebase grep]

Phase 1.5 should not rewrite the legacy thread API. The current thread surface still serializes only literal payloads and explicitly rejects functions because closures may reference thread-local state, while the approved actor design says Phase 2 will replace that serialize-round-trip with tiered send semantics and use Phase 1.5 as the gate for shared closure handlers. (`src/gene/serdes.nim:765-782`, `src/gene/vm/thread_native.nim:293-301`, `docs/proposals/actor-design.md:80-115`, `docs/proposals/actor-design.md:711-715`) [VERIFIED: codebase grep] [CITED: docs/proposals/actor-design.md]

The main planning risk is that closures are only partially productized today: nested/local functions already capture scope and execute, but regression coverage still comments out returned-closure tests as "closures not fully implemented". That makes Phase 1.5 an extension of a real runtime path, but not one with complete behavioral coverage yet. (`tests/integration/test_function_optimization.nim:99-105`, `tests/integration/test_function_optimization.nim:119-130`, `tests/integration/test_function_optimization.nim:159-166`) [VERIFIED: codebase grep]

**Primary recommendation:** Freeze `VkFunction` by walking its captured `Scope` graph with the same two-pass validate-then-tag discipline used by Phase 1 containers, leave `serialize_literal` and thread transport unchanged, and make Phase 2 consume a new closure-freezeability predicate instead of rediscovering closure rules in the actor send path. [VERIFIED: codebase grep] [CITED: docs/proposals/actor-design.md]

## Architecture Patterns

### System Architecture Diagram

```text
Closure Value (VkFunction)
  -> read Function.parent_scope + ScopeTracker
  -> validate captured Scope graph
     -> immediate / already-frozen values pass
     -> Phase 1 freezable containers recurse
     -> capability / actor-local values fail with typed path
  -> tag deep_frozen/shared on closure + captured managed graph
  -> publish through existing shared-bit invariant
  -> Phase 2 actor send checks deep_frozen
     -> yes: pointer-share closure O(1)
     -> no: clone-or-reject path handled later by actor send design
```

### Pattern 1: Two-Pass Closure Freeze
**What:** Extend the existing Phase 1 `validate_for_freeze` / `tag_for_freeze` pattern so `VkFunction` validates its captured environment first and only then marks the closure and reachable managed captures as `deep_frozen` and `shared`. (`src/gene/stdlib/freeze.nim:46-140`) [VERIFIED: codebase grep]

**When to use:** Any time a closure is the value being frozen, or appears inside a larger graph that a caller passes to `(freeze v)`. [CITED: docs/proposals/actor-design.md]

**Example:**
```nim
# Source basis: src/gene/stdlib/freeze.nim, src/gene/vm/exec.nim, src/gene/types/core/collections.nim
proc validate_for_freeze(v: Value, path: string, visitedVals, visitedScopes) =
  case v.kind
  of VkFunction:
    validate_function_scope(v.ref.fn, path & "/<env>", visitedVals, visitedScopes)
  else:
    validate_existing_phase1_kinds(v, path, visitedVals)

proc tag_for_freeze(v: Value, visitedVals, visitedScopes) =
  case v.kind
  of VkFunction:
    setDeepFrozen(v)
    setShared(v)
    tag_function_scope(v.ref.fn, visitedVals, visitedScopes)
  else:
    tag_existing_phase1_kinds(v, visitedVals)
```

### Pattern 2: Freeze the Captured Scope Graph, Not the Invocation Frame
**What:** Closure freezability should derive from `Function.parent_scope` and the values reachable from `Scope.members`, because frames are pooled execution objects while scopes are the capture boundary. (`src/gene/types/type_defs.nim:623`, `src/gene/types/type_defs.nim:1178-1196`, `src/gene/types/core/collections.nim:142-176`, `src/gene/types/core/frames.nim:58-71`) [VERIFIED: codebase grep]

**When to use:** Any design decision about "what is the closure environment?" or "what object gets walked for captured values?" [VERIFIED: codebase grep]

**Example:**
```nim
# Source basis: src/gene/vm/exec.nim:3424-3452 and src/gene/types/core/collections.nim:172-176
let env = fn.parent_scope
if env != nil:
  for captured in env.members:
    validate_for_freeze(captured, path, visitedVals)
```

## Common Pitfalls

### Pitfall 1: Treating Function Code as the Whole Closure
**What goes wrong:** The planner assumes closures are immutable because function code is immutable, then forgets that captured values live outside the bytecode body. (`src/gene/types/type_defs.nim:621-627`, `src/gene/vm/exec.nim:3424-3429`) [VERIFIED: codebase grep]
**Why it happens:** The runtime stores closure capture on `parent_scope`, not in `body_compiled` or the matcher. (`src/gene/types/type_defs.nim:622-623`) [VERIFIED: codebase grep]
**How to avoid:** Make the freeze rule about the captured `Scope` graph first, and only then set `deep_frozen/shared` on the function value. [VERIFIED: codebase grep]
**Warning signs:** A design doc that only mentions `body_compiled`, `native_entry`, or bytecode publication when describing closure sendability. [VERIFIED: codebase grep]

### Pitfall 2: Accidentally Expanding Phase 1.5 into Actor Send or Thread Rework
**What goes wrong:** The implementation starts touching `Thread.send`, `serialize_literal`, or scheduler code before closure-freezeability itself is settled. (`src/gene/vm/thread_native.nim:257-317`, `src/gene/serdes.nim:765-782`) [VERIFIED: codebase grep]
**Why it happens:** The actor design talks about closure sendability and cloneability in the same conceptual section, but the roadmap still makes Phase 1.5 a single-requirement prerequisite. (`docs/proposals/actor-design.md:552-556`, `.planning/ROADMAP.md:82-93`) [CITED: docs/proposals/actor-design.md] [VERIFIED: codebase grep]
**How to avoid:** Plan Phase 1.5 around closure-freeze validation/tagging plus tests, and leave send routing or clone implementation to Phase 2 unless a helper API is required. [CITED: docs/proposals/actor-design.md] [ASSUMED]
**Warning signs:** Tasks mentioning mailbox code, actor queues, or `serialize_literal` before any task mentions captured-scope traversal. [VERIFIED: codebase grep]

### Pitfall 3: Freezing Frames Instead of Scopes
**What goes wrong:** A proposal walks `FrameObj` or `caller_frame` and bakes execution-state pointers into a supposedly shareable closure. (`src/gene/types/type_defs.nim:1178-1196`) [VERIFIED: codebase grep]
**Why it happens:** Blocks capture frames, and some call paths borrow scope directly when matchers are empty. (`src/gene/types/type_defs.nim:640-648`, `src/gene/vm/exec.nim:1730-1737`) [VERIFIED: codebase grep]
**How to avoid:** Limit Phase 1.5 to `VkFunction` closure capture over `Scope`; explicitly defer `VkBlock` until a concrete use case proves it is needed. [VERIFIED: codebase grep] [CITED: docs/proposals/actor-design.md] [ASSUMED]
**Warning signs:** Design notes that talk about preserving caller chain, macro caller context, or frame stack state for a frozen closure. [VERIFIED: codebase grep]

### Pitfall 4: Missing Path-Rich Errors for Non-Freezable Captures
**What goes wrong:** `(freeze closure)` fails generically with "function not sendable", which is not good enough for planning or later actor debugging. [CITED: docs/proposals/actor-design.md] [ASSUMED]
**Why it happens:** Current `FreezeScopeError` path reporting is container-centric and does not yet name closure capture slots. (`src/gene/stdlib/freeze.nim:23-31`, `src/gene/stdlib/freeze.nim:57-77`) [VERIFIED: codebase grep]
**How to avoid:** Reuse the existing path format and introduce a stable closure segment such as `/<closure>/<capture:N>` or a symbol-based capture name when available. [ASSUMED]
**Warning signs:** Test assertions check only the outer `VkFunction` kind and cannot identify which captured value blocked freezing. [VERIFIED: codebase grep]

2. **Does Phase 1.5 need mutable-closure cloneability, or only frozen-closure pointer-sharing?**  
   What we know: the requirement only demands freezing and pointer-sharing, while the actor design later discusses cloneable unfrozen closures. (`.planning/REQUIREMENTS.md:59-61`, `docs/proposals/actor-design.md:552-556`) [VERIFIED: codebase grep] [CITED: docs/proposals/actor-design.md]
