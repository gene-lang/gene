<objective>
Lock the stable-core semantic contracts for `nil` versus `void`, selectors,
Gene expression evaluation, macro input shape, and the tested pattern-matching
subset with public docs and runnable tests.
</objective>

<threat_model>
Primary risk: public docs overclaim semantics that are not actually enforced,
causing users to build DSLs, selector-heavy code, or optional-value flows on
undefined behavior.

Mitigations:
- Treat tests as the arbiter for every Phase 06 semantic claim.
- Prefer documentation plus focused regression tests before source changes.
- Only change VM/compiler/selector/matcher source when a new focused test proves
  a real mismatch against the chosen contract.
- Keep pattern matching, ADTs, and selector update breadth explicitly bounded;
  do not redesign them in this phase.
</threat_model>

<must_haves>
- `nil` and `void` are defined as distinct observable outcomes in public specs.
- Selector docs and selector tests agree on missing key/index, nil receiver,
  defaults, strict access, stream modes, and `$set` update boundaries.
- Gene expression docs explain type/callee, properties, children, normal call
  evaluation, macro input shape, `$caller_eval`, and metadata boundaries.
- Pattern docs separate a tested stable subset from experimental/future gaps.
- Focused Nim tests run for all semantic surfaces changed by this phase.
</must_haves>

    Add a public value contract that states:
    - `nil` is an explicit value for intentional absence and optional values.
    - `void` is the result of missing member/index/match lookup and is observable by tests/printing, but is not the ordinary optional-value result.
    - Missing map keys, missing object/instance properties, missing Gene properties, and out-of-range array/Gene child indices return `void`.
    - Lookup on a `nil` receiver propagates `nil`.
    - Defaults replace `void` lookup failure, not explicit `nil`.
    - A `case` expression with no matching `when` and no `else` returns `nil`.
    - Function bodies return their last expression, so optional-style functions should return `nil` explicitly when they mean "no value".

    Create `tests/integration/test_core_semantics.nim` with focused `test_vm`
    coverage for:
    - map missing key returns `VOID`; map key present with `nil` returns `NIL`.
    - array out-of-range returns `VOID`; array element present with `nil` returns `NIL`.
    - Gene missing property and child out-of-range return `VOID`; Gene property present with `nil` returns `NIL`.
    - instance missing property returns `VOID`; instance property present with `nil` returns `NIL`.
    - `case` without a matching branch returns `NIL`.
    - a function returning explicit `nil` returns `NIL`.

    Add `nim c -r tests/integration/test_core_semantics.nim` to the
    `testintegration` task in `gene.nimble` near the other core integration
    tests.
  </action>

    target states above. Include explicit tests that first-class selector calls

    explicit `nil`.

    likely fixes:

      a `NIL` base return `NIL`, not `VOID`.
    - in `src/gene/vm/exec.nim` and `src/gene/stdlib/selectors.nim`, make
      default values replace `VOID` lookup failure but not explicit `NIL`.
  </action>

    - `$caller_eval` evaluates the AST value in the caller's scope;
    - `$render` renders a quoted/template value before evaluation.

    argument with both a property and children using `.type`, `.props`, and
